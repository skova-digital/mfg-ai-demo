# 検査記録AI デモ — 現場の報告を、そのまま品質データに

**🔗 動くデモ: https://mfg-ai-demo.kf-works-contact.workers.dev** （「ダッシュボード」から見るのがおすすめです）

> **本デモは架空の業務設定と生成データ（seed 固定の決定論的ダミーデータ）によるものです。実在の企業・製品・人物・実データとは一切関係ありません。**

中小製造業の現場では、検査記録や作業報告が紙のまま運用されていることが今も珍しくありません（本デモの課題設定は業界で広く聞かれる一般的なものであり、特定企業の事例ではありません）。紙運用には共通の弊害があります。

- 伝票が現物に1枚しか付かないため、**その紙を持っている人しか最新情報を追えない**。何かあったときの確認は特定の担当者に集中する
- 検査内容への追記や備考は、**伝票が新しくなるたびに手で書き写す**。転記のたびに漏れ・誤記のリスクが乗る
- 過去の伝票は箱の山。**過去実績を探すのは発掘作業**になり、similar な不良が前にもあったかを誰も即答できない
- 集計されないので、**不良がどの品番・どの工程に偏っているかが見えない**。気づいたときには社外流出している

このデモは、**現場の雑な報告文や帳票の内容を AI がその場で構造化データに変換し、蓄積すると不良傾向がダッシュボードで見えてくる**——という一連の流れを、実際に動く形で示すものです。

## 何が楽になるか（Before → After）

| 場面 | Before（紙運用） | After（このデモの方式） |
|------|----------------|----------------------|
| 記録 | 検査後に伝票へ手書き→転記 | 口語のまま入力（または帳票を撮影）→ AI が構造化・人は確認だけ |
| 情報共有 | 伝票を持つ人しか最新を知らない | 保存した瞬間から全員が同じデータを参照できる |
| 過去実績 | 箱の山から発掘 | 品番・工程・期間で即検索 |
| 傾向把握 | 集計されず、勘と記憶 | 不良率の推移・偏りがダッシュボードに自動で出る |

## 何が見られるか（3ステップ）

1. **入力**: 現場の口語そのままの報告（例:「MK-1024のロットB、切削上がり50個みたけど2個バリ。うち1個は限度見本ギリ」）や帳票写真を投入
2. **AI 構造化 → 人が確定**: Claude API が品番・ロット・数量・不良分類・判定を抽出。自信のない項目はハイライトされ、**人が確認・修正してから保存**される（AI は下書き係、確定は人——現場に受け入れられるための設計判断です）
3. **ダッシュボード**: 蓄積データから不良率の推移・不良分類の内訳・品番×工程の偏りが見える。同梱のダミーデータには「ある品番のある工程だけ不良率が悪化している」状況を意図的に仕込んであり、開いた瞬間に「気づける」ことを体験できます

保存した記録は**記録一覧**ページで品番・工程・期間・判定から即座に検索できます（紙の伝票の山からの発掘作業がなくなる、の実演です）。

## アーキテクチャ

```
[ブラウザ]  入力(テキスト/写真) → 確認・修正 → ダッシュボード
    │ fetch
[Cloudflare Workers]  /api/extract ─→ Claude API (tool use で JSON スキーマ強制)
    │                 /api/records, /api/stats
[Cloudflare D1]  inspection_records（検査記録）
```

- **依存最小**: フロントはビルドなしの素の HTML/CSS/JS、グラフは SVG 手書き、バックエンドは Workers 1ファイル
- **構造化の安定性**: Claude の tool use（structured output）でスキーマ準拠を強制。自由文からの抽出でも型が崩れない
- **デモモード**: 公開版は事前用意した AI 変換結果を再生する `EXTRACT_MODE=sample` で動作（API 費用ゼロで恒久公開できる設計）。ライブ変換はローカル実行手順で確認できます

## セキュリティとデータの扱いについて

「AI に業務データを渡して大丈夫なのか」という質問は導入検討で必ず出るため、このデモでも設計段階から次の前提を置いています。

- **本デモに実データは一切含まれません**。全レコードは生成スクリプトによる架空データです
- **API キーはサーバー側（Workers Secrets / ローカルは `.dev.vars`）でのみ保持**し、ブラウザに露出しません。リポジトリにも含まれません
- **AI に送るのは構造化に必要な報告テキストのみ**。蓄積データベース（D1）の中身を AI に渡す設計にはしていません
- Anthropic の商用 API は、送信データをモデルの学習に使用しないことがポリシーとして明示されています（実導入時は最新の公式ポリシーと社内規程の確認を前提とします）
- 全 API 入力にスキーマ検証・サイズ上限・enum 制約を実装しています（`src/worker.js`）
- 実運用に進む場合の追加検討事項（認証・アクセス制御・監査ログ・保持期間）は本デモのスコープ外として意図的に省いています

## ローカルで動かす

```bash
npm install -g wrangler   # 未導入の場合
node scripts/gen-dummy.mjs                     # ダミーデータ生成（seed 固定・再現可能）
wrangler d1 execute mfg_ai_demo --local --file=scripts/schema.sql
wrangler d1 execute mfg_ai_demo --local --file=scripts/seed-data.sql
cp .dev.vars.example .dev.vars                 # ライブ変換を試す場合は API キーを設定
wrangler dev                                    # http://localhost:8787
```

`.dev.vars` を設定しない場合も、`EXTRACT_MODE=sample`（既定）でデモ全体が動作します。

## 設計判断メモ

- **なぜ「人が確定」を挟むか**: 検査記録は品質保証の根拠になるデータであり、AI の抽出ミスがそのまま記録になる設計は現場で信頼されない。AI の役割を「転記の下書き」に限定し、確定権を人に残すことで、導入障壁と誤記録リスクを同時に下げる
- **なぜ Workers + D1 か**: サーバー常時起動が不要な業務デモでは、従量課金のエッジ実行が費用面で合理的（本デモの想定費用はゼロ円/月）
- **なぜダミーデータに「物語」を仕込むか**: ダッシュボードの価値は「集計が見える」ことではなく「異常に気づける」こと。それを口で説明せず、開いた瞬間に体験してもらうため
